- 版本：v1.1（方案稿，修订）
- 日期：2026-03-26（首版） / 2026-03-27（修订：路由分区、Header、身份模型、视觉对齐）
- 适用范围：Web App（Next.js + Supabase + tailwindcss + tailwind-merge）
- 说明：本文档仅覆盖第一版范围内可执行事项，待确认后进入编码阶段。

## 1. 目标与约束

### 1.1 方案目标
- 将 `docs/AIText.html` 原型转化为可维护的 Next.js App Router 结构。
- 严格按产品主文档第一版范围实施，避免范围蔓延。
- 页面组件化、业务逻辑 Hook 化、数据库调用统一收敛至 API Router。
- 产出可直接执行的 Supabase 数据库设计（独立 SQL 文档）。

### 1.2 已有能力（不重做）
- 账号体系（登录、注册、忘记密码、修改密码）已集成，沿用现有流程。
- AI 实时对话（RTC）已集成，关键代码：
  - `src/hooks/useRtcTutorSession.ts`
  - `src/app/api/realtime/session/route.ts`
- 本期不重构账号系统，仅对任务场景可选增强 RTC 上下文传参。

### 1.3 第一版范围冻结
- 做：
  - 组织、成员、角色（Owner/Member）
  - 目标（Goal）管理
  - 任务（Task）管理：创建、领取、完成、重复任务
  - 排行榜、总览、近期活动
  - 成员与群组管理、邀请链接入组
  - AI 辅助生成任务（文本/文档 -> 建议任务 -> 批量导入）
  - 站内通知（基础版）
- 不做：
  - WhatsApp 集成（仅预留）
  - 积分商城
  - 任务审核流
  - 原生 App、多语言、复杂 BI

## 2. 技术实施架构

### 2.1 技术栈
- 前端：Next.js（App Router）+ React + TypeScript
- 样式：Tailwind CSS + tailwind-merge
- 数据层：Supabase（PostgreSQL + RLS）
- 服务层：Next.js API Router（`src/app/api/**/route.ts`）

### 2.2 分层原则（强约束）
- UI 层（page/components）：
  - 只负责展示和交互触发。
- Hooks 层（`src/hooks/**`）：
  - 处理业务状态、请求编排、缓存与乐观更新。
- API Router 层（`src/app/api/**`）：
  - 统一进行 Supabase 读写与权限校验。
- 禁止：
  - 页面组件直接访问 Supabase。
  - hooks 直接写数据库。

### 2.3 路由分区（Owner 与 Member 物理隔离）
- **目标**：Owner 工作台与 Member 工作台分目录管理，降低误把管理页暴露给成员的维护成本；**业务组件可跨区复用**。
- **Owner（组织拥有者）**：页面一律放在 `src/app/(admin)/…` 下（与现有 `admin` 分区一致时，TaskAI Owner 路由可共用 `(admin)` layout，或通过子路径区分，如 `/admin/taskai/…`，具体 URL 在编码阶段与现有后台路由对齐，但**物理目录锚定在 `(admin)`**）。
- **Member（组织成员）**：页面放在 `src/app/(main)/…` 下。
- **共用**：任务卡片、状态标签、排行榜列表、活动流等放入 `src/components/taskai/`（或现有 `components` 下独立命名空间），由 `(admin)` 与 `(main)` 页面按需引用。
- **进入规则**：
  - 登录后根据当前用户在某组织下的 `organization_memberships.role` 决定可访问的路由树；若用户仅为 Member，**禁止访问** `(admin)` 下 TaskAI Owner 路由（layout 或 middleware 重定向）。
  - 若用户既是 A 组织 Owner 又是 B 组织 Member，切换组织后应重新判定入口（仍不出现 HTML 原型中的「角色下拉切换」）。

## 3. 页面与组件拆分方案

### 3.1 页面路由（分区建议）

**Member —— `src/app/(main)/…`（示例路径，可待 URL 命名统一）**
- `/taskai/tasks`（或 `/tasks`）：成员任务看板（认领、我的进行中、与我相关的已完成）
- `/taskai/overview`：成员总览（简化 KPI、个人排名、目标贡献等）
- `/taskai/partners`：伙伴 / 群组成员视图
- `/taskai/settings` 或复用现有 `/(main)/my/settings`：个人设置（与全局账号设置衔接）
- `/taskai/notifications`：站内通知（可选）

**Owner —— `src/app/(admin)/…`**
- `/admin/taskai/tasks`（示例）：Owner 任务管理（创建、编辑、看板/列表、AI 导入）
- `/admin/taskai/overview`：Owner 总览（完整洞察卡片）
- `/admin/taskai/members`：成员与群组、邀请链接
- `/admin/taskai/settings`：组织级设置（积分池、组织信息，可与个人设置拆分）

> 路径前缀仅为示例；**强制约束**是「Owner 页仅存在于 `(admin)` tree」「Member 页仅存在于 `(main)` tree」。

### 3.2 顶部栏（TopBar）与 `Header.tsx` 复用
- **不再单独实现**名为 `TopBar` 的全量顶栏；**以 `src/components/layout/Header.tsx` 为统一顶栏**。
- **改造要求**（编码阶段执行）：
  - 支持可配置 **主标题**（替代当前写死的 `TalentSync AI`），例如 props：`title?: string`、`subtitle?: string`。
  - TaskAI 场景下标题与 `AIText.html` 一致：**`Task` + 高亮 `AI`**（可用 `Task` + `<span className="text-indigo-600">AI</span>`），副标题可为 `Intelligent Collaboration` 或与产品文案一致的一行说明。
  - 可选：`showBrandIcon`、右侧插槽（积分池、组织切换器、通知铃铛）以 props 或 `children` 传入，避免把 TaskAI 逻辑写死在 Header 内部。
  - Owner/Member 各自 layout 传入不同 `title`/`subtitle`/右侧区块即可；**同一组件、不同配置**。

### 3.3 身份与数据可见性（取代原型里的角色/成员下拉）
- **删除** `AIText.html` 中 `roleSwitch`（Owner/Staff 下拉）与 `staffSelect`（切换当前成员）的交互模式；**产品上不提供「模拟他人身份」**。
- **真实行为**：
  - 当前登录用户即唯一身份；所有列表、任务、积分、排行榜均以 **`auth.uid()` 对应的 membership** 为边界。
  - Member 仅请求「对自己可见」的任务（无群组则全员可见；有 `task_visible_groups` 则按所属群组过滤），**只能认领/完成与自己相关的任务**。
  - Owner 在 `(admin)` 内看到组织级管理数据；API 与 RLS 双重保证成员不可写管理接口。
- **与 SQL 对齐**：`docs/db/taskai_v1_schema.sql` 中 RLS 以 `auth.uid()` 与 `organization_memberships` 为准；前端不再维护 `currentStaffId` 类状态。

### 3.4 视觉规范：与 `AIText.html` 对齐
以下为 V1 UI 还原的**硬约束**（实现时用 Tailwind 等价类，必要时抽成小组件内联 class 或 `tailwind-merge` 合并）：

- **页面背景**：`bg-slate-50 min-h-screen`（与原型 body 一致）。
- **任务卡片容器**：`card-hover bg-white rounded-2xl border border-slate-200 p-4 shadow-sm`，hover：`hover:-translate-y-1`（或原型 `card-hover`：translateY + 阴影），进场可保留 `fade-in-up` 及递增 `animation-delay`（可用 CSS module / globals 或 framer-motion，**视觉效果接近即可**）。
- **状态标签（Status）**：
  - Open：`bg-blue-100 text-blue-700 text-xs font-bold px-2.5 py-0.5 rounded-full`
  - In Progress：`bg-amber-100 text-amber-700`（同级字号与圆角）
  - Completed：`bg-emerald-100 text-emerald-700`
- **类型标签（One-time / Recurring）**：
  - Recurring：`bg-purple-100 text-purple-700` + `fa-sync-alt` 语义图标（可用 `lucide-react` 等价图标保持尺寸一致）
  - One-time：`bg-slate-100 text-slate-600` + 闪电类图标
- **积分行**：`fa-star text-amber-400` + `font-bold text-slate-700` 数值 + `text-slate-400 text-xs` 的 `pts` 文案层级。
- **主按钮（认领等）**：`bg-indigo-600 hover:bg-indigo-700 text-white text-sm font-semibold py-2 rounded-xl`。
- **「Work with AI」按钮**：原型为 `from-violet-600 to-indigo-600` 渐变条，需保留渐变与 `rounded-xl`、`shadow-lg shadow-violet-200` 气质一致。

实现前可对 `AIText.html` 中 `renderTaskCard`、`renderHeader`、`renderTaskBoard` 等片段做一次 class 摘录表，编码时逐条对照。

### 3.6 HTML -> Next.js 迁移映射
- `Task Board` -> `(main)` / `(admin)` 各自 `tasks/page.tsx` + 共用 `TaskCard` 系列组件。
- `Leaderboard` -> `overview/page.tsx`（数据源按 Member/Owner 区分）+ `LeaderboardList`。
- `Team Management` -> **仅** `(admin)/…/members/page.tsx`。
- `Organization Insights` -> **仅** `(admin)/…/overview/page.tsx`（只保留 V1 指标；不上线 HTML 中纯演示的 ROI / 虚构 AI 效率数字，与产品主文档一致）。

- `renderHeader` 中的品牌区 -> 由扩展后的 `Header.tsx` 承载（§3.2），**不包含**原型中的 `roleSwitch` / `staffSelect`（§3.3）。

## 4. 数据与接口方案（概要）

> 详细 SQL 见：`docs/db/taskai_v1_schema.sql`

### 4.1 核心数据实体
- `profiles`（用户扩展信息）
- `organizations`
- `organization_memberships`
- `goals`
- `tasks`
- `task_claims`
- `groups`
- `group_memberships`
- `task_visible_groups`
- `activities`
- `notifications`
- `invite_links`
- `ai_task_generation_jobs`
- `ai_task_suggestions`

### 4.2 API Router 规划（V1）
- 组织与成员：
  - `GET/POST /api/orgs`
  - `GET/PATCH /api/orgs/[orgId]`
  - `GET /api/orgs/[orgId]/members`
- 目标与任务：
  - `GET/POST /api/goals`
  - `PATCH/DELETE /api/goals/[id]`
  - `GET/POST /api/tasks`
  - `PATCH/DELETE /api/tasks/[id]`
  - `POST /api/tasks/[id]/claim`
  - `POST /api/tasks/[id]/complete`
- 总览与榜单：
  - `GET /api/overview?orgId=...`
  - `GET /api/leaderboard?orgId=...&groupId=...`
  - `GET /api/activities?orgId=...`
- 群组与邀请：
  - `GET/POST /api/groups`
  - `PATCH/DELETE /api/groups/[id]`
  - `POST /api/invites`
  - `POST /api/invites/accept`
- AI 任务生成：
  - `POST /api/ai/tasks/generate`
  - `POST /api/ai/tasks/import`

## 5. Hooks 设计（业务逻辑统一收口）

- `useCurrentOrg()`
- `useTaskBoard({ orgId, goalId, status })`（`role` 由服务端 membership 判定，hooks 仅区分 Owner API vs Member API 的调用编排，**不提供切换身份的参数**）
- `useTaskActions()`
- `useGoals(orgId)`
- `useOverview(orgId)`（同上，视图差异由路由 `(admin)`/`(main)` 与接口返回字段决定）
- `useLeaderboard(orgId, groupId?)`
- `useMembers(orgId)`
- `useGroups(orgId)`
- `useInviteLinks(orgId)`
- `useNotifications(orgId)`
- `useAiTaskGenerator(orgId)`

## 6. RTC 能力衔接（仅必要调整）

### 6.1 保持不变
- 现有 RTC 会话创建与结束流程保持不变，不影响当前功能稳定性。

### 6.2 可选增强（V1 内轻量）
- 在创建 RTC 会话时增加可选上下文字段：
  - `taskId`
  - `taskTitle`
  - `taskHints`
- 目的：在 “Work with AI” 任务场景中让对话更聚焦。
- 原则：不改现有核心 RTC 生命周期与鉴权逻辑。

## 8. 验收标准（确认后执行）

- 功能验收：
  - Owner 可创建目标与任务；Member 可领取并完成可见任务。
  - 重复任务可多次完成并保留历史记录。
  - 排行榜按组织/群组可正确计算。
  - **无** Owner/Member 或「切换为某成员」的调试控件；登录用户仅能操作自己的数据边界内的行为。
- 架构验收：
  - 所有数据库调用均位于 API Router。
  - 所有页面业务状态由 hooks 管理。
  - 组件拆分可复用，页面结构清晰。
  - **Owner 页面仅存在于 `src/app/(admin)`；Member 页面仅存在于 `src/app/(main)`**（共用组件除外）。
  - TaskAI 相关顶栏基于 **`Header.tsx` 扩展**，标题/副标题可配置，TaskAI 场景下与原型品牌呈现一致。
- UI 验收：
  - 任务卡片、状态标签、类型标签、主按钮与 `AIText.html` 的 Tailwind 规格一致或肉眼同层级（见 §3.4）。
- 安全验收：
  - Supabase RLS 生效，Member 不可越权管理组织资源。
