1. 这次我更新了什么

这次更新主要集中在 TaskAI 主流程、WhatsApp 绑定 / 通知、以及 Vercel 上线后的配置文件与部署说明。

1.1 TaskAI 主介面与数据加载

  • TaskAI 主要页面在 refresh / hydration 时，先显示 loading，不再先闪 mock data 或错误空状态。
  • member / admin 端的 org selection 初始化逻辑已收敛成共用 hook。
  • 主要 TaskAI hooks 已补上 request 去重与 stale response 保护，减少 React 开发环境或初次加载时的重复请求与旧响应覆盖。
  • admin insights 里原本偏 marketing / mock 的区块已改成 live organization data。

1.2 WhatsApp 绑定与通知

  • 用户现在可以在设置页绑定 WhatsApp 电话，并接收验证码完成验证。
  • 设置页会显示 verification 状态、通知偏好、最近通知记录与 scheduled time。
  • 已加入以下通知事件：
    • task_new_available
    • task_claimed
    • task_claimed_no_ai_started
    • task_claimed_stalled
    • task_completed_encourage
    • binding_verification_code
  • admin 端新增 /admin/taskai/notifications，可以查看通知发放记录。
  • 系统支持用户解除绑定，解除时会取消尚未完成的 verification。

1.3 本轮不在修改范围内的部分

  • Dify
  • transcribe
  • AI 对话主链路

这些部分这次没有继续改动，避免影响现有流程。

2. 这次部署后最重要的事情

重点：push 到 Vercel 并不代表正式站的 WhatsApp 会自动打到 Bobby 本机。要让正式站通知真正发送，除了 Vercel 部署完成，还必须把本机 dispatcher 的目标地址切到正式域名，并确保 bridge token 一致。

当前架构是：

Vercel / 正式站建立 notification jobs
        ↓
本机 Mac dispatcher 主动 pull /api/internal/whatsapp/jobs/claim
        ↓
openclaw-whatsapp 在本机真正发送 WhatsApp
        ↓
再把发送结果回写到正式站

也就是说：

  • 不是 Vercel 主动打进本机。
  • 而是 Bobby 本机主动去正式站拉 job。
  • 如果本机还在拉 http://localhost:3000，那正式站 job 不会被这台 Mac 处理。

3. 阿聪部署时要做的事

3.1 拉最新代码

  • GitHub branch：main
  • 最新 commit：8a14451e8ea4281958f0b12fbda0d2b720aa3a18

3.2 在 Vercel Production 配环境变量

至少要确认这些变量存在且正确：

NEXT_PUBLIC_SUPABASE_URL=...
NEXT_PUBLIC_SUPABASE_ANON=...
SUPABASE_SERVICE_ROLE_KEY=...

NEXT_PUBLIC_SITE_URL=https://正式域名
NEXT_PUBLIC_BASE_URL=https://正式域名

RTC_APP_ID=...
RTC_APP_KEY=...
VOLC_ACCESS_KEY_ID=...
VOLC_SECRET_ACCESS_KEY=...
VOLC_S2S_APP_ID=...
VOLC_S2S_ACCESS_TOKEN=...

ARK_API_KEY=...
ARK_MODEL_ID=...

TASKAI_INTERNAL_BRIDGE_TOKEN=与本机 dispatcher 完全一致

如果正式站仍要走 Bobby 本机 WhatsApp：TASKAI_INTERNAL_BRIDGE_TOKEN 一定要配，而且这个值必须和 Bobby 本机的 /Users/bobbylian/.taskai-whatsapp-dispatch.env 里的值完全一致。

3.3 Bobby 本机要同步做的配置

这部分通常不是阿聪在 Vercel 上做，而是 Bobby 本机要同步确认：

TASKAI_BASE_URL=https://正式域名
TASKAI_INTERNAL_BRIDGE_TOKEN=与 Vercel Production 相同
OPENCLAW_WHATSAPP_ADDR=http://127.0.0.1:8555

如果 TASKAI_BASE_URL 没改，仍然是 http://localhost:3000，那正式站通知不会真正发出去。

4. 部署完成后怎么验收

  1. 确认 Vercel 最新部署成功。
  2. 打开正式站，确认 TaskAI 页面能正常进入，且不再先闪假资料。
  3. 进入 /my/settings，确认 WhatsApp 区块显示正常。
  4. 如果已绑定 WhatsApp，测试 Send test message。
  5. 检查收到的 WhatsApp 文案链接域名是否为当前正式域名。
  6. 如果要验证本机 worker：
    • 检查 openclaw-whatsapp status --addr http://127.0.0.1:8555 是否为 connected
    • 检查 launchctl list | rg ai.openclaw.taskai-whatsapp-dispatch
    • 确认 job 最终会从 queued -> sending -> sent

7. 关于 env，要不要发给阿聪

结论：不要把你整份 .env.local 原封不动发给他。

正确做法是：

  1. 把 .env.example 发给他，作为变量清单。
  2. 真正需要的生产 secrets，再用安全方式单独发：
    • Vercel Dashboard 直接配置
    • 1Password / 私密密码管理工具
    • 私下安全讯息，不要丢在多人群组
  3. 如果阿聪只负责 Vercel 部署，他通常不需要你本机所有变量。
  4. 本机专用的东西，例如 OPENCLAW_WHATSAPP_ADDR、本机路径、LaunchAgent 相关信息，不需要整包交给他，除非他也要接手你这台 Mac 的本机 worker。

最少应该单独、安全提供给阿聪的，通常是：

  • Vercel 生产环境需要的 keys
  • TASKAI_INTERNAL_BRIDGE_TOKEN（如果他要配 production WhatsApp bridge）
  • 当前正式域名应该设为什么
