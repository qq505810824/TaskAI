# RTC / 对话录音存储与上传性能方案（对象存储 + URL 落库）

**文档日期**：2026-03-24  
**状态**：可行性说明（本文档不包含代码实现）  
**背景**：当前学生侧与 AI 侧录音常以 **Data URL（base64）** 形式写入 `conversations.user_audio_url` / `ai_audio_url`，经 `POST /api/conversations/batch` 等大 JSON 请求写入 **Supabase（Postgres）**。单条可达 **百 KB～数百 KB** 时，请求体大、序列化/网络/数据库写入耗时明显，拖慢「结束会议」等流程体验。

---

## 一、问题归纳

| 现象 | 原因 |
|------|------|
| 结束会议 / 批量保存变慢 | HTTP 请求体巨大（多轮对话 × 两段音频 Data URL） |
| 数据库压力大 | 大文本字段频繁写入、行宽膨胀、备份与复制成本上升 |
| 前端/网关超时风险 | 移动网络下大 POST 更易失败或重试 |
| 回顾页加载潜在变慢 | 若列表接口仍返回完整 Data URL，首屏需下载大量内联数据 |

**结论**：**不宜长期将音频二进制以 base64 内嵌在 Postgres 行内**；宜改为 **对象存储存文件 + 库中仅存 URL（或 storage path）**。

---

## 二、目标对齐

1. **文件存到专门存储，库中只存 URL（或 path）** — 批量保存请求体显著变小，上传时间主要变为「多路小文件上传」或「后台异步上传」，可与用户感知解耦。  
2. **不改变原有对话过程与实时体验** — RTC 信令、字幕、本地播放逻辑可保持不变；仅调整 **持久化时机与载体**（何时上传、上传到哪里）。  
3. **上传与获取都要「够快」** — 需从 **链路、并发、区域、格式** 多维度优化（见下文）。  
4. **可演进** — 支持渐进迁移、双写/读兼容，避免一次性大改上线风险。

---

## 三、推荐总体架构（首选）

### 3.1 存储选型（与 Supabase 生态契合）

- **Supabase Storage（S3 兼容）**  
  - **优点**：与现有项目同一账号与区域、权限模型（RLS + policies）、Signed URL 成熟。  
  - **用法**：Bucket 存 `wav` / `webm` / `opus` 等文件；表字段存 **`https://...` 公开 URL** 或 **`bucket/path`** + 需要时再签名的私有访问。

- **备选**：Cloudflare R2、阿里云 OSS、AWS S3、火山 TOS 等 — 若已有 CDN/合规/成本策略，可统一走 **同一对象存储 + CDN**。

### 3.2 上传路径（两种可行模式）

**模式 A：客户端直传 Storage（推荐减轻 Next 服务器压力）**

1. 结束一轮对话或结束会议前，前端已持有 `Blob` / `ArrayBuffer`。  
2. 调用 **受鉴权的 Next API**：`POST /api/storage/upload-session`（命名示例），返回 **Signed Upload URL** 或 **Supabase client 用用户 JWT 上传**（视安全模型而定）。  
3. 客户端 **PUT/POST 到 Storage**，成功后得到 **object path** 或 **永久/长期公开 URL**。  
4. `conversations` 里只写 **短字符串 URL 或 path**，再 `batch` 插入 — 请求体从数百 KB 降至 KB 级。

**模式 B：服务端中转（实现简单，瓶颈在应用服务器）**

1. 客户端仍可把较小 payload 发到 Next（或分片）。  
2. 服务端写入 Storage，再把 URL 写入 DB。  
3. 适合早期 PoC；**高并发或大文件**时易占满 Node 内存与带宽，不如模式 A。

### 3.3 与「不影响对话体验」的衔接

| 环节 | 建议 |
|------|------|
| **实时对话中** | 不强制每轮立刻上传完整文件；可仅在 **内存 / 本地 Blob URL** 中保持播放与字幕逻辑。 |
| **结束会议** | **关键路径**：先 `stop RTC`、保证字幕与文本已就绪 → **可先 `batch` 写入文本 + 占位或空音频字段** → 后台队列上传音频并 **PATCH 更新 URL**；或 **并行**：小文本 batch 与上传并行，UI 显示「正在同步录音…」。 |
| **必须同步完成才离开** | 若产品要求「离开页面前一定落库完整」：可对上传做 **并发限制（如 2～3 路）** + **超时降级**（先存 path 占位，失败标记 `upload_failed` 供重试）。 |

**原则**：**实时链路与持久化链路解耦** — 用户听到的声音不依赖 Storage 上传完成；落库慢不应阻塞 RTC 已建立的会话。

## 四、其他有效手段（速度与体验）

### 4.1 体积与格式

- **优先压缩格式**：在兼容前提下，**Opus in WebM / OGG** 往往比 **WAV + PCM** 小一个数量级，上传与下载更快。  
- **采样率/码率**：语音场景可适当降低采样率（需与 ASR/回放质量平衡）。  
- **分段存储**：极长会话可按「每轮一条 object」而非合并超大单文件，利于并行上传与断点重试。

### 4.2 网络与区域

- Storage Bucket 与 **主要用户区域**、**Next/Supabase 区域** 一致，降低 RTT。  
- 对回放域名启用 **CDN**（Storage 自带或前置 CDN），提升 **获取速度**。

### 4.3 上传策略

- **并发上限**：避免浏览器同时开 10+ 连接导致排队与失败。  
- **重试与幂等**：object key 含 `user_meet_id + conversation_id + side(user|ai)`，重复上传覆盖或跳过。  
- **后台上传**：使用 `requestIdleCallback` / `queueMicrotask` / `worker` 仅在 **交互空闲** 时传，减少主线程卡顿（对 UI 影响小）。

### 4.4 数据库与 API

- **batch 接口只接收 URL 字符串**，拒绝或截断超大 Data URL（可在网关层限制 body size）。  
- 列表/摘要接口（如 `meet-summary`、`GET /api/meets/[id]`）**默认不返回** 音频 URL 或仅返回 **短链/path**，详情页再按需加载。

### 4.5 安全

- **私有 Bucket + Signed URL**（限时读取），避免长期公开链接泄露。  
- 上传策略校验：`meet_id` / `user_meet_id` 与当前用户身份匹配，防止越权写他人路径。

## 五、迁移与兼容（可行性）

1. **阶段 1**：新写入一律上传 Storage，字段存 URL；读逻辑同时支持 **以 `http`/`https` 开头** 与 **legacy Data URL**。  
2. **阶段 2**：离线脚本或按需任务将历史 Data URL **解码后写入 Storage**，再把行内字段替换为 URL（注意批处理与失败重试）。  
3. **阶段 3**：库表可加 `user_audio_storage_path` 等显式列（可选），或统一约定 `user_audio_url` 语义为「任意可访问 URI」。

## 六、风险与注意点

- **Supabase / Postgres 单行与 JSON 请求体限制**：超大 base64 易触顶或导致偶发失败。  
- **Data URL 在日志与监控中的泄露**：改为 URL 后注意日志脱敏。  
- **结束会议总时长**：若坚持「全部上传完才跳转」，需明确 **超时与重试 UI**；更推荐 **弱阻塞 + 后台补齐**。  

---

## 七、小结建议（可执行优先级）

| 优先级 | 动作 |
|--------|------|
| P0 | 对象存储存文件，`conversations` 仅存 **URL 或 path**；`batch` 请求体缩小。 |
| P1 | **客户端直传 Signed URL**（或 Supabase 客户端上传），Next 只做鉴权与元数据。 |
| P2 | 音频格式向 **压缩容器** 演进；结束会议流程 **异步上传** 或与文本 batch **并行**。 |
| P3 | 列表/摘要接口 **默认剥离** 大字段；回放页 **按需拉取** 音频 URL。 |

## 八、按当前需求的定稿方案（本轮确认）

本节用于落实你提出的 4 条优化，作为后续实现的明确基线。

### 8.1 先压缩 base64 对应音频，再上传

- 录音文件在上传前先做**压缩/转码**（优先 `webm/opus`，不建议继续 `wav` 原始 PCM 直传）。  
- 如当前链路先得到 Data URL，可先转回 `Blob`，再压缩后上传；不再把压缩后的 base64写进数据库。  
- 目标：显著降低上传体积，减少移动网络下耗时与失败率。

### 8.2 仅保留学生录音：`user_audio_url`

- 存储策略调整为：**只上传学生音频**，并只在 `conversations` 写 `user_audio_url`（URL/path）。  
- `ai_audio_url` 不再作为持久化目标（可在实时会话中临时播放，但不写库/不回传）。  
- 好处：减少约一半音频上传量，降低存储与带宽成本。

### 8.3 结束会议提交时，移除 `ai_audio_url`

- 在结束会议提交 `conversations` 前，对 payload 做规范化：  
  - 删除/置空 `ai_audio_url`；  
  - 保留 `user_audio_url`（已上传后的 URL/path）与必要文本字段。  
- 原则：**batch 只接收文本 + 轻量元数据 + 学生音频 URL**，不再携带 AI 音频内容。

### 8.4 首阶段实施路径：Supabase Storage + batch 只写 URL

- **首阶段直接采用**：Supabase Storage（对象存储）+ `conversations/batch` 仅写 URL。  
- 推荐流程：
  1. 前端拿到学生录音 Blob；  
  2. 压缩/转码；  
  3. 上传 Supabase Storage，得到 `public URL` 或 `storage path`；  
  4. `POST /api/conversations/batch` 时仅提交 `user_audio_url=<url/path>`；  
  5. 不提交 `ai_audio_url`。  
- 这样可在不改变 RTC 主交互体验的前提下，显著缩短结束会议提交耗时。

### 8.5 字段语义建议（避免后续歧义）

- `user_audio_url`：统一语义为 **对象存储地址（URL/path）**。  
- `ai_audio_url`：保留字段用于兼容历史数据，但新写入默认 `null/空`。  
- 读接口默认不返回大字段，回放页按需获取（已有方向保持不变）。
