TaskAI — 產品主文件
版本： v0.1 草案
最後更新： 2026 年 3 月 26 日
撰寫目的： 供內部團隊閱讀，統一對產品方向、功能範圍與設計原則的理解，作為後續開發、設計與 AI 整合的唯一真實來源（Single Source of Truth）。

一、產品願景與背景
TaskAI 是一個面向組織（公司、社團、課堂、小型團隊等）的任務管理與激勵平台。我們的核心理念是：把「完成任務」這件事變得像遊戲一樣有回饋感，同時讓組織擁有者能清楚看見每位成員的貢獻與組織整體的健康度。
市面上不缺任務管理工具，但大多數工具存在兩個問題：第一，成員缺乏動力去主動領取並完成任務；第二，組織擁有者缺乏一個簡單的方式來量化「任務完成」對組織目標的實際推進。TaskAI 試圖同時解決這兩個問題——透過積分制與排行榜創造內在動力，透過目標與任務的關聯讓擁有者看到全局。
此外，我們深知不是每個人都會主動打開一個網頁去查看任務。因此 TaskAI 的 AI 助理將透過 WhatsApp 與 Telegram 主動觸及成員，在他們最熟悉的通訊軟體中推送提醒、提供資訊、引導他們回到平台完成任務。這不是一個聊天機器人——它是一個有節制的「任務管家」。

二、我們的設計約束（Design Constraints）
在進入功能細節之前，所有參與開發和設計的同事必須牢記以下幾點，這些約束將貫穿整份文件。
我們是新創，資源有限。 這意味著每個功能都必須證明自己的存在價值。如果一個功能「有了很好，但沒有也不會死」，那它就不應該出現在第一版。我們寧可把五個功能做到好用且穩定，也不要做十五個半成品。
使用者不應該需要學習。 我們的使用者可能是學生、可能是小公司員工，他們不會閱讀使用手冊。因此介面必須使用他們已經熟悉的 UI 模式——例如類似 Instagram 的底部導航列、類似 Notion 的卡片式佈局、類似 WhatsApp 的聊天介面。我們不發明新的交互方式，我們借用已被驗證的模式。
功能應有機合併，而非無限展開。 如果「排行榜」和「組織洞察」都是擁有者關心的資訊，那它們就應該共存於同一個畫面中，而非分成兩個獨立頁面。每多一個頁面、每多一個按鈕，都是認知負擔。我們追求的是「一眼就懂」。
AI 必須省錢。 我們的 AI 整合（WhatsApp / Telegram bot）每一次對話都有 token 成本。因此 bot 的設計不是「什麼都能聊」，而是「快速完成特定任務後結束對話」。它是一個有禮貌的管家，不是一個閒聊夥伴。

三、使用者角色定義
整個平台只有兩種角色，沒有第三種。簡單就是力量。
擁有者（Owner） 是建立組織的人。他制定目標、建立任務、管理成員、查看全局數據。一個組織只有一個擁有者（未來可能開放共同管理者，但第一版不做）。擁有者也可以是成員——例如一個小團隊的負責人，自己也會去完成任務——但在介面上他看到的是「擁有者視角」。
成員（Staff / Member） 是被加入組織的人。他看到可領取的任務、自己的進度、排行榜、以及他所屬的群組和隊友。成員不能建立任務，不能修改組織設定，不能看到某些擁有者專屬的統計數據（例如全員的詳細生產力分析）。
一個人可以同時是 A 組織的擁有者和 B 組織的成員。平台應支援多組織切換，但第一版的 UI 可以先做得簡單——在設定頁或頂部有一個組織切換器即可。

四、資訊架構（Information Architecture）
以下是整個平台的頁面結構。我們用底部導航列（Bottom Navigation Bar）作為主要導航方式，因為這是手機使用者最熟悉的模式。擁有者和成員看到的導航列略有不同。
擁有者看到的底部導航列有三個分頁：
第一個分頁叫做「任務」。這是擁有者建立目標、建立任務、管理任務的地方。第二個分頁叫做「總覽」。這是擁有者查看排行榜、組織目標進度、各種統計數據、以及近期活動的地方——也就是「Dashboard」的概念，但我們刻意不使用 Dashboard 這個詞，因為「總覽」更直覺。第三個分頁叫做「成員」。這是擁有者管理組織成員、建立群組、邀請新人的地方。
成員看到的底部導航列也有三個分頁：
第一個分頁叫做「任務」。成員在這裡看到所有可領取的任務、自己正在進行的任務、以及已完成的任務。第二個分頁叫做「總覽」。成員在這裡看到排行榜、自己的積分與成就、以及組織的近期動態。注意，成員看到的「總覽」比擁有者的簡化很多——沒有組織健康度指標、沒有部門貢獻分析、沒有 ROI 計算。第三個分頁叫做「夥伴」。成員在這裡看到自己的隊友、所屬群組、以及組織內的其他成員。底部導航列只有三個按鈕。三個。不多不少。這是刻意的。

五、核心概念模型（Core Concepts）
在深入各頁面之前，我們需要先定義平台中幾個核心概念之間的關係，所有人必須對此有一致的理解。
組織（Organization） 是最上層的容器。一個組織有一個擁有者、若干成員、若干目標、若干任務。所有資料都歸屬於組織。
目標（Goal） 是組織層級的方向性聲明。例如「完成產品 v2.0 上線」或「本季客戶滿意度提升 15%」。目標有名稱、描述、以及目標積分值（即達成此目標預期需要的總積分）。目標的進度由其下的任務完成情況自動計算，擁有者不需要手動更新。
任務（Task） 是最小的工作單位，永遠歸屬於某個目標之下。任務有名稱、描述、積分值、類型（一次性或重複性）、狀態（開放、進行中、已完成）、以及分類標籤。任務是成員實際去「做」的東西。一個目標下可以有很多任務。
積分（Points） 是整個激勵系統的貨幣。擁有者在建立任務時設定積分值，成員完成任務後獲得對應積分。積分會反映在排行榜上。組織有一個「積分池（Pool）」的概念，代表組織預計發放的總積分——這可以幫助擁有者做預算管理，但第一版中這是選填功能，不強制使用。
群組（Group） 是成員的集合。擁有者可以建立群組，例如「工程部」、「設計部」、「A 班」。群組的用途有二：第一，擁有者可以將任務指派給特定群組，只有該群組的成員能看到並領取；第二，排行榜和統計數據可以按群組篩選。群組是選填功能——如果組織很小，可以完全不建群組，所有成員看到所有任務。
AI 助理（AI Assistant） 不是一個獨立的頁面或功能，而是一個貫穿整個平台的輔助角色。在網頁端，AI 協助擁有者從文件中生成任務。在 WhatsApp / Telegram 端，AI 主動聯繫成員，提醒任務、推送排行榜、引導成員回到平台。AI 助理沒有自己的「頁面」，它存在於各個功能的上下文中。

六、擁有者功能詳述
6.1 分頁一：任務
這個頁面是擁有者建立和管理工作的主要場所。頁面頂部是組織的目標列表，下方是任務列表。兩者的關係是：點選一個目標，任務列表會自動篩選為該目標下的任務；不選擇任何目標，則顯示所有任務。這不需要兩個頁面——目標以水平滑動的卡片或膠囊按鈕（pill-shaped filter）呈現在頁面頂部，任務以垂直列表呈現在下方。
目標管理： 擁有者可以新增目標。新增目標時需填寫：目標名稱（必填）、目標描述（選填）、目標圖示（從預設圖示中選擇，選填）、目標積分值（選填，若填寫則會顯示進度百分比）。目標建立後會出現在頂部的篩選列中。擁有者可以編輯和刪除目標。刪除目標時，其下的任務不會被刪除，而是變成「無歸屬目標」的任務——但我們應該在 UI 上提示擁有者，建議他同時處理這些任務。
任務建立（手動）： 擁有者可以手動建立任務。建立任務時需填寫：任務名稱（必填）、任務描述（選填）、所屬目標（必填，從已建立的目標中選擇）、積分值（必填）、類型（一次性或重複性，必填。若為重複性，需額外選擇頻率：每日、每週、每月）、分類標籤（選填，從預設標籤或自訂標籤中選擇）、可見群組（選填，若不選則所有成員可見）。
任務建立（AI 輔助）： 這是我們的差異化功能之一。擁有者可以上傳一份文件（PDF、Word、純文字），或直接貼上一段文字，AI 會根據內容自動生成建議的任務清單。例如，擁有者上傳了一份「Q1 專案計畫書」，AI 會解析文件內容，產出五到十個建議任務，每個任務都有預填的名稱、描述、建議積分值和建議歸屬目標。擁有者可以逐一檢視，修改後確認採用，或直接刪除不需要的。這個功能在 UI 上的呈現方式是：任務建立表單旁邊有一個「AI 生成」按鈕。點擊後彈出一個對話框，擁有者在其中上傳文件或貼上文字，AI 處理後返回建議列表，擁有者勾選想要的任務後一鍵匯入。整個流程不超過三步。
任務管理： 每張任務卡片上會顯示：任務名稱、積分值、類型標籤（一次性/重複性）、狀態標籤（開放/進行中/已完成）、分類標籤、以及目前認領的成員頭像（若有）。擁有者可以點擊任務卡片展開詳情，在詳情中可以編輯、刪除、或手動改變任務狀態。任務列表預設按狀態排序：進行中的在最上面，開放的次之，已完成的在最下面。擁有者也可以切換為看板模式（Kanban view），以三欄呈現：開放、進行中、已完成——但列表模式是預設，因為在手機上看板模式的體驗較差。
重複性任務的機制： 重複性任務（例如「每週程式碼審查」）在被完成後不會變成「已完成」狀態，而是自動重置為「開放」狀態，等待下一次認領。每次完成都會獨立計算積分。擁有者可以在任務詳情中看到該重複性任務的歷史完成紀錄——誰在什麼時間完成了多少次。

6.2 分頁二：總覽
這個頁面是擁有者的全局視圖，所有與「組織現在怎麼樣了」相關的資訊都集中在這裡。頁面採用上下滾動的卡片式佈局，每一張卡片是一個資訊模組。擁有者向下滾動即可看到越來越深入的數據——從最重要的概要到較細節的分析，排列順序是刻意設計的。
頂部概要列（Summary Bar）： 頁面最頂部是四個小型數據卡片，橫向排列（手機上為兩行兩列）。分別顯示：總任務數（含趨勢箭頭，例如「較上週 +12%」）、已完成任務數、已發放積分數、目前活躍成員數（即正在進行任務的人數）。這些數字讓擁有者在一秒鐘內掌握組織脈搏。
組織目標進度卡片： 緊接在概要列下方。每個目標以一張卡片呈現，顯示：目標名稱、目標描述、進度條（已完成任務積分 / 目標積分值）、百分比、以及「X/Y 任務已完成」的文字。如果擁有者建立了四個目標，這裡就會有四張卡片，以兩列的網格呈現。這讓擁有者能直覺地看到「任務的完成如何推動了組織的大方向」。
排行榜卡片： 顯示前五名成員的積分排名。以簡潔的列表形式呈現——頭像、名稱、積分、排名。前三名有獎牌圖示。如果擁有者想看完整排行榜，可以點擊「查看全部」展開完整列表（以 Modal 或頁內展開方式，不跳轉新頁面）。排行榜可以用頂部的膠囊按鈕篩選群組——例如只看「工程部」的排名。
任務完成率卡片： 這張卡片包含一個圓環圖，顯示整體完成率百分比，旁邊列出三個數字：已完成、進行中、開放。下方有一行小字顯示「平均每位成員 X 分」。
每週活動卡片： 一個簡單的長條圖，X 軸是本週一到日，Y 軸是當天全組織所獲得的積分。如果擁有者建立了群組，長條圖上方有群組篩選器，可以切換為查看單一群組的週活動。這讓擁有者看到「哪些天團隊比較活躍，哪些天比較安靜」。
近期活動卡片： 一個時間線（Timeline）式的列表，顯示最近的事件。事件類型包括：某人完成了某任務（含獲得積分）、某人認領了某任務、新任務被建立、新成員加入組織。每條事件有頭像、文字描述、時間戳記。這個列表預設顯示最近十條，向下滾動可載入更多。
以上所有卡片存在於同一個可滾動的頁面中。沒有子頁面、沒有跳轉。擁有者打開「總覽」分頁，向下滾動，就能依序看到所有資訊。

6.3 分頁三：成員
這個頁面管理「誰在你的組織裡」。
成員列表： 頁面主體是組織所有成員的列表。每個成員的列項顯示：頭像、名稱、積分、目前狀態（空閒/正在進行某任務）、所屬群組標籤。列表頂部有搜尋框和群組篩選器。
邀請成員： 頁面右上角有一個「邀請」按鈕。點擊後生成一個邀請連結（URL），擁有者可以複製此連結並透過任何管道（WhatsApp、Email、口頭）分享給想邀請的人。被邀請者點擊連結後，會進入簡單的註冊或登入流程，完成後自動加入該組織。我們不做複雜的邀請系統（例如輸入 Email 批量寄送邀請信）——一條連結就夠了。如果擁有者希望連結有時效或人數限制，可以在進階設定中調整，但第一版直接做成永久連結即可。
群組管理： 頁面中有一個切換，讓擁有者在「所有成員」和「群組檢視」之間切換。群組檢視以卡片呈現每個群組——群組名稱、成員數量、群組成員頭像（最多顯示五個）。點擊群組卡片可以進入群組詳情，在其中可以：修改群組名稱、新增或移除成員（從已加入組織的成員中選擇）、刪除群組。建立新群組只需填寫群組名稱，然後勾選要加入的成員。
成員詳情： 在成員列表中點擊某位成員，會以側邊抽屜（Drawer）或 Modal 的方式顯示該成員的詳細資訊：積分總數、已完成的任務列表、目前正在進行的任務、所屬群組、加入組織的時間。擁有者也可以在這裡將該成員從組織中移除。

七、成員功能詳述
7.1 分頁一：任務
成員看到的任務頁面和擁有者的類似，但權限不同。頁面頂部同樣有目標篩選器（成員也能看到組織的目標，這有助於他們理解「我的工作如何與大方向連結」）。下方是任務列表。
可見的任務： 成員只能看到對他可見的任務——即未指定群組的公開任務，以及指定給他所屬群組的任務。成員看不到指定給其他群組的任務。
認領任務： 每張開放狀態的任務卡片上有一個「認領」按鈕。成員點擊認領後，任務狀態變為「進行中」，並標記該成員為執行者。一個成員在同一時間可以認領幾個任務——這個限制由擁有者在組織設定中決定。第一版的預設是不限制，但 UI 上應預留此設定的空間。
完成任務： 成員在自己正在進行的任務卡片上有一個「標記完成」按鈕。點擊後，任務狀態變為「已完成」，成員獲得對應積分。是否需要擁有者審核才能算完成？第一版的設計是不需要——成員自行標記完成即可獲得積分。未來可以加入「需審核」選項作為擁有者可配置的功能，但第一版保持簡單。完成任務時，UI 應有明確的慶祝回饋——例如撒花動畫、積分彈出動畫——這對激勵感非常重要。
任務列表的分段： 成員的任務列表預設分為三個區段，以分隔線和標題區分：「我正在做的」（進行中且我是執行者）排在最上面；「可認領的」（開放狀態）排在中間；「已完成的」（我完成的歷史任務）排在最下面。這樣成員打開任務頁面時，第一眼看到的就是自己目前該做的事。

7.2 分頁二：總覽
成員的總覽頁面比擁有者的簡化許多，但核心精神相同——讓成員快速了解「我在組織中的位置」和「大家最近在幹嘛」。
個人狀態卡片： 頁面最頂部是一張突出的個人狀態卡片，顯示：成員自己的頭像、名稱、總積分、目前排名（例如「第 3 名，共 12 人」）、以及目前正在進行的任務名稱（若有）。這張卡片的設計應該讓成員感受到「這是關於我的」。
排行榜卡片： 和擁有者看到的相同——前五名成員的排名，前三名有獎牌，可展開查看全部。如果組織有群組，也可以按群組篩選。成員自己的排名在列表中會被高亮顯示，不管他排第幾。

組織目標進度卡片（簡化版）： 成員也可以看到組織目標的進度。但和擁有者不同的是，成員看到的版本額外標示了「我對這個目標貢獻了 X 分」——讓成員感受到自己的工作如何推動了組織大方向。這是非常重要的激勵設計。
近期活動卡片： 和擁有者看到的相同，但成員可能會更關注和自己相關的事件，因此在設計上可以考慮用不同的背景色標示「與我相關」的事件（例如有人完成了我也有認領的任務、或我所屬群組的動態）。
成員的總覽頁面不包含：組織洞察數據卡片（總任務數、完成率等屬於管理層資訊）、每週活動長條圖（此為擁有者用於管理分析的工具）、部門貢獻分析、ROI 摘要。

7.3 分頁三：夥伴
這個頁面讓成員看到「和我一起在這個組織裡的人是誰」以及「我的隊友是誰」。
群組檢視： 頁面頂部顯示成員所屬的群組。每個群組以卡片呈現——群組名稱、成員頭像、群組成員數量。點擊群組卡片可以看到群組內的所有成員列表，以及該群組的排行榜。這創造了一種「團隊歸屬感」。
組織成員列表： 群組卡片下方是整個組織的成員列表（簡化版），顯示每個人的頭像、名稱、積分。成員可以看到其他人，但不能看到其他人正在進行什麼任務的詳情——只能看到「忙碌中」或「空閒」的狀態。這保護了一定的隱私，同時又讓成員知道「誰在做事」。
成員不能管理群組（不能建立、編輯、刪除群組），也不能邀請新成員加入組織。這些都是擁有者的權限。

八、AI 整合（WhatsApp / Telegram Bot）
這是 TaskAI 最具差異化的功能之一，但也是最需要謹慎控制成本的功能。以下詳述 AI 助理在通訊平台上的行為規範。
8.1 設計哲學
AI 助理的定位是「任務管家」，不是「聊天夥伴」。它的每一次發言都有明確的目的：推動成員去完成任務。它不閒聊、不回答與任務無關的問題、不進行開放式對話。這不是因為技術上做不到，而是因為每一次 AI 回覆都消耗 token，而 token 就是金錢。
核心原則是：AI 主動推送，被動回應有限。 也就是說，AI 會在特定時機主動傳訊息給成員（推送），但成員想跟 AI 聊天的能力是受限的（有限回應）。
8.2 AI 主動推送的情境
閒置提醒： 如果一位成員目前沒有任何進行中的任務，且超過一定時間（例如 24 小時）沒有認領新任務，AI 會傳送一條訊息：「嗨 [名字]！你目前沒有正在進行的任務。現在有 [X] 個任務等你認領，最高可獲得 [Y] 積分。[查看任務（連結）]」。這條訊息簡短、有行動呼籲（CTA）、包含連結。訊息中不包含任務的完整清單——我們要把成員引導回平台，而不是在通訊軟體中重建整個任務列表。
進度提醒： 如果一位成員認領了任務但超過一定時間（例如 48 小時）沒有標記完成，AI 會傳送一條訊息：「嗨 [名字]，你正在進行的「[任務名稱]」已經進行了 [X] 天。需要幫助嗎？完成後可獲得 [Y] 積分。[查看任務（連結）]」。
排行榜推送： 每週一次（例如每週一早上），AI 會向所有成員傳送本週排行榜概要：「📊 本週排行榜更新！第一名：[名字] [積分]分，第二名：[名字] [積分]分，第三名：[名字] [積分]分。你目前排名第 [X] 名。繼續加油！[查看完整排行榜（連結）]」。
任務完成恭喜： 當成員完成一個任務時，AI 會立即傳送一條訊息：「🎉 太棒了 [名字]！你完成了「[任務名稱]」，獲得 [Y] 積分。你現在的排名是第 [X] 名。[查看排行榜（連結）]」。
新任務通知： 當擁有者建立了新任務且該成員有權看到時，AI 會傳送一條訊息：「📋 新任務：「[任務名稱]」—— [簡短描述]，完成可獲得 [Y] 積分。[立即認領（連結）]」。如果同時建立了多個任務（例如 AI 輔助生成後一次匯入五個），應合併為一條訊息：「📋 [X] 個新任務剛剛發佈！最高可獲得 [Y] 積分。[查看所有新任務（連結）]」。
8.3 AI 被動回應的設計
當成員在通訊軟體中主動對 AI 傳訊息時，AI 只會回應以下幾類詢問，其餘一律禮貌拒絕。
查詢我的狀態： 成員可以說類似「我的積分是多少」、「我現在排第幾」、「我在做什麼任務」之類的話。AI 回覆簡短的事實資訊加上連結。例如：「你目前有 280 積分，排名第 3。你正在進行「每週程式碼審查」。[查看詳情（連結）]」。
查詢可用任務： 成員可以說「有什麼任務可以做」。AI 回覆最多三個開放任務的簡要資訊，加上連結。不顯示全部——引導回平台。
查詢排行榜： 成員可以說「排行榜」。AI 回覆前五名和成員自己的排名。
其他所有對話： AI 回覆：「我是 TaskAI 的任務管家，專門負責任務相關的事務。如果你需要查看任務、積分或排行榜，隨時問我！如果有其他問題，請聯繫你的組織管理者。」然後停止。不繼續對話。不嘗試理解成員在說什麼。一條訊息結束。

8.4 Token 成本控制策略
每次 AI 被動回應，系統計算消耗的 token 數量。每位成員每天有被動詢問次數上限——第一版設定為五次。超過後 AI 回覆：「你今天的詢問額度已用完，明天再來！你可以隨時上平台查看最新資訊。[前往 TaskAI（連結）]」。
AI 主動推送不計入成員的次數上限，因為推送的內容是預先生成的模板，token 消耗極低（大部分是填入變數的固定文字）。
為進一步控制成本，所有主動推送訊息應盡量使用模板化的文字，只在必要時呼叫 AI 模型。例如，排行榜推送完全不需要 AI 模型——只需要從資料庫讀取排名數據，填入模板即可。只有「AI 輔助生成任務」功能需要真正呼叫大型語言模型。

8.5 成員綁定流程
成員如何將自己的 WhatsApp 或 Telegram 帳號和 TaskAI 帳號綁定？在平台的設定頁面中，成員點擊「連結 WhatsApp」或「連結 Telegram」，系統產生一個一次性的驗證碼。成員在 WhatsApp / Telegram 中找到 TaskAI 的 bot，傳送該驗證碼，bot 驗證後完成綁定。此後 bot 就知道這個通訊帳號對應的是哪位成員。
擁有者不需要為每個成員手動設定——這是成員自助完成的。擁有者只需要告訴成員「去設定頁面連結你的通訊帳號」。

九、積分系統與激勵設計
積分系統是 TaskAI 的心臟。如果積分沒有意義，整個平台的激勵機制就會崩塌。因此我們需要在設計上賦予積分足夠的「重量感」。
積分的來源： 成員獲得積分的唯一途徑是完成任務。沒有其他來源。不設登入獎勵、不設簽到獎勵、不設邀請獎勵。簡單乾淨——你做事，你得分。
積分的可見性： 積分在以下地方可見——排行榜（全組織和群組）、成員的個人狀態卡片、任務卡片上的積分標示、擁有者的總覽頁面。積分是公開的，所有成員都能看到彼此的積分。這是刻意的——公開排名創造良性競爭。
積分的用途（第一版）： 第一版中，積分本身沒有「兌換」功能。它純粹是一個榮譽指標，用於排行榜排名。這是刻意簡化的——我們先驗證「積分排名是否足以驅動成員行為」，再考慮是否需要加入兌換商城等功能。但在資料結構上，積分應記錄為可累加也可消費的餘額，以便未來擴展。
組織積分池（選填功能）： 擁有者可以設定一個組織積分池的總量——例如「本季總共 5000 積分」。每當任務被完成，積分從池中扣除。這幫助擁有者做資源規劃——他不會不小心建立超過預算的任務。如果擁有者不設定積分池，所有任務的積分就是無上限發放。

十、全平台共用元素
頂部導航列（Top Bar）： 所有頁面共用一個頂部導航列，包含：左側的 TaskAI 品牌標誌、右側的組織切換器（若使用者加入了多個組織）、以及使用者頭像（點擊後進入設定頁面）。在擁有者視角中，頂部也顯示積分池餘額（若有設定）。
設定頁面： 從使用者頭像進入。包含：個人資料編輯（名稱、頭像）、通訊帳號綁定（WhatsApp / Telegram）、通知偏好（是否接收 AI 推送、推送頻率）、登出。對擁有者而言，設定頁面額外包含：組織資訊編輯（組織名稱、描述）、積分池設定、邀請連結管理、以及危險操作區（解散組織）。
通知系統： 平台本身也有站內通知。在頂部導航列的使用者頭像旁有一個通知鈴鐺圖示，點擊展開通知列表。通知類型和 AI 推送的情境相同——新任務、任務完成、排名變動等。站內通知確保即使成員沒有綁定通訊帳號，也不會錯過重要資訊。
空白狀態（Empty States）： 每個列表都需要設計空白狀態——例如，當組織還沒有任何任務時，任務頁面不能是一片空白，而應顯示一段引導文字和一個「建立第一個任務」的按鈕（擁有者）或「等待擁有者發佈任務」的說明（成員）。空白狀態的設計對新使用者的第一印象至關重要。
載入狀態與動畫： 所有資料載入時應有 Skeleton Loading 動畫（灰色閃動的佔位區塊），不顯示空白頁面或旋轉圖示。任務卡片的進場應有微妙的淡入動畫（fade-in-up），但不過度——動畫是為了讓介面感覺流暢，不是為了炫技。

十一、技術與平台備註
此章節為工程團隊提供方向性指引。
平台優先級： TaskAI 的第一版是 Web App（響應式網頁應用），同時支援桌面和手機瀏覽器。不做原生 App（iOS / Android）。未來若需要，可考慮用 PWA 或 React Native 包裝。
AI 輔助生成任務的技術路徑： 文件上傳後，後端將文件內容提取為純文字，送入 LLM（例如 Claude 或 GPT），搭配精心設計的 prompt，要求模型回傳結構化的 JSON 格式（任務名稱、描述、建議積分、建議歸屬目標）。前端接收 JSON 後渲染為可編輯的任務建議卡片。需注意：文件可能很長，需做分段摘要再生成任務，以控制 token 用量。
WhatsApp / Telegram 整合的技術路徑： Telegram 使用官方 Bot API，免費且穩定。WhatsApp 使用 WhatsApp Business API（需透過官方合作夥伴或 Meta Cloud API），有一定成本但覆蓋面廣。兩個平台共用同一套「訊息策略引擎」——該引擎決定何時對誰發送什麼訊息，然後透過對應平台的 API 發出。Bot 接收成員訊息後，先做意圖分類（可以用簡單的關鍵字比對 + 輕量模型），確認是否為支援的詢問類型，再決定是回覆事實資訊還是禮貌拒絕。
資料模型核心實體（供工程團隊參考）： 系統包含以下核心實體——Organization（組織）、User（使用者）、Membership（使用者與組織的關聯，包含角色、積分等）、Goal（目標，歸屬於組織）、Task（任務，歸屬於目標）、TaskAssignment（成員與任務的關聯，記錄認領時間、完成時間等）、Group（群組，歸屬於組織）、GroupMembership（成員與群組的關聯）、Activity（活動記錄，用於近期活動和通知）。

十二、第一版範圍與不做的事
以下明確列出第一版會做的和不會做的，避免範圍蠕變。
第一版會做的： 擁有者建立組織、建立目標、建立任務（手動 + AI 輔助）、管理成員和群組、查看總覽數據。成員認領任務、完成任務、獲得積分、查看排行榜和個人狀態、查看夥伴和群組。AI 在 Telegram 上做主動推送和有限被動回應。網頁端的基本通知系統。
第一版不做的： WhatsApp 整合（先做 Telegram，WhatsApp 的審批流程較長，第二版再加）。積分兌換商城。任務完成的擁有者審核機制。成員之間的私訊功能。任務的檔案附件上傳。任務的截止日期（第一版所有任務都是開放式的，做完即可）。多語系。原生手機 App。自訂主題或品牌顏色。

十三、開放問題與待決事項
以下問題尚未有結論，需團隊討論後決定。
第一，成員自行標記任務完成是否足夠？是否需要加入某種簡單的驗證機制——例如完成時需上傳截圖或填寫簡短的完成報告？這會增加摩擦力但也增加可信度。
第二，一個任務是否可以被多人認領？目前設計是一個任務只能被一人認領。但某些情境下可能需要多人協作。如果開放多人認領，積分如何分配？
第三，群組之間是否可以競爭？例如顯示「群組排行榜」——按群組的總積分排名。這可能會增加團隊競爭感，但也可能造成不公平（人多的群組天然佔優勢，除非用人均積分）。
第四，AI 推送的頻率如何拿捏？太頻繁會讓成員感到煩躁，太少又失去提醒的效果。第一版先設定固定頻率（閒置提醒每 24 小時一次、進度提醒每 48 小時一次），未來可根據數據調整。
第五，是否需要擁有者儀表板中的「AI 影響力指標」（如 AI 效率、節省時間、節省成本）？這些數字在產品早期很難準確計算，放上去可能顯得空洞。建議第一版不做，等累積足夠數據後再加。
